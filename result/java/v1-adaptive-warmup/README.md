# Java v1 적응형 warmup 결과

측정일: 2026-06-20
실행 환경: 루트 [`README.md`](../../../README.md#실행-환경) 기준, **Rust 베이스라인과 동일 환경** (OCI, aarch64 Neoverse-N1, 2 vCPU, Ubuntu 24.04.4 LTS) — RSS 컬럼이 채워진 것으로 Linux 측정임이 확인됨
실행 명령: `./gradlew bench` (`java-engine/`)
측정 방식/CSV 컬럼 정의: [`java-engine/src/main/java/dev/junyoung/bench/README.md`](../../../java-engine/src/main/java/dev/junyoung/bench/README.md)
비교 대상: [`Rust 엔진 기준선`](../../rust-baseline/), 이전 버전: [`v0-initial`](../v0-initial/)

> [v0-initial](../v0-initial/)에서는 고정 1-pass 워밍업 때문에 `run_0`/`run_1`이 JIT 컴파일 비용을
> 떠안아 mean이 steady-state보다 낮게 나왔다. 본 버전은 `CompilationMXBean` 컴파일 시간 델타가
> 연속 2회 0ms가 될 때까지 워밍업 pass를 반복하는 적응형 워밍업을 도입했다(최대 10패스 cap).
> 워밍업 pass도 처리량뿐 아니라 레이턴시/진단 코드 경로까지 실행하도록 확장했다.

## 관찰

### 워밍업 효과는 명확하지만 시나리오마다 다르게 나타난다
- **WorstCaseCross/1M이 86%→103%로 가장 크게 개선**됐다. v0에서는 `run_0`이 GC(17.2회)와 잔여 JIT
  컴파일에 눌려 mean을 끌어내렸는데, 본 버전은 `jit_compile_ms`가 run 전체에서 0~1ms로 균일해
  steady-state에 가까운 값이 처음부터 나온다. 그 결과 Java가 Rust를 **3% 앞선다**(이전 v0의
  WorstCaseCross/10M 역전과 같은 성격).
- **ActiveFill/10M도 69%→73%로 개선**됐다. 반면 **ThinBook/10M(49%→47%)과 ActiveFill/1M(85%→82%)은
  소폭 하락**했는데, `jit_compile_ms` mean이 이미 1~3ms로 낮아 워밍업 여유는 충분했고 stddev가 커
  (예: ActiveFill/1M stddev 1,785,209 vs v0의 2,877,457 — 오히려 줄었음) 이 정도 변동은 run-to-run
  노이즈로 보는 것이 타당하다. 결론적으로 적응형 워밍업은 **JIT 미완료로 인한 하향 편향을 제거**하지만
  GC·메모리 대역폭이 지배하는 구간(ThinBook 대량 삽입, ActiveFill 대규모 체결)의 격차까지는 줄이지 못한다.

### 진단 신호: jit_compile_ms가 사실상 0에 수렴
- 모든 시나리오·스케일의 `jit_compile_ms` mean이 **0.4~2.8ms**로, v0-initial(0.5~29.7ms, 특히
  ActiveFill/1M의 run_0=283ms)과 비교해 한 자릿수 이상 낮고 run 간 분산도 작다. 적응형 워밍업이
  의도한 대로 컴파일 비용을 측정 구간 밖으로 밀어냈다는 직접 증거다.
- 다만 완전히 0은 아니다(`ThinBook/1M` run_1의 jit=16ms, 같은 run의 p999=1,280ns·max=2,018,652ns로
  유독 큰 max). 비동기 컴파일 스레드가 측정 도중에도 드물게 재트리거될 수 있다는 한계로 본다(워밍업
  README의 "근사값" 설명과 일치).

### 메모리: heap_used가 v0보다 전반적으로 커졌다
- 모든 시나리오의 `heap_used_peak/avg`가 v0-initial 대비 커졌다(예: ActiveFill/1M peak 1.47M→1.22M은
  비슷하나 avg 1.38M→1.04M처럼 혼재). 이는 워밍업 pass 자체가 레이턴시/진단 경로까지 실행하도록
  확장되면서 워밍업 중 할당된 객체가 본 측정 전 힙에 더 많이 누적된 영향으로 보인다. RSS 배수
  자체는 v0과 거의 동일(ThinBook 2.2×/1.6×, ActiveFill 6.4×/9.2×, WorstCaseCross 24.8×/64.6×)해
  워밍업 확장이 RSS 신호를 왜곡하지는 않았다.

### 여전히 남아있는 꼬리 레이턴시 이상치
- `ActiveFill/10M`의 max stddev가 34.6M ns로 거대하다 — `run_6`(원본 CSV) 단독으로 max=115,477,409ns
  (115ms)를 기록해 평균을 끌어올린다. 다른 run은 32k~182k ns대다. 워밍업은 JIT만 다루므로 GC
  세이프포인트로 인한 단발성 스톨은 그대로 남아있다.
- `WorstCaseCross/10M`도 `run_3/7/8`의 max가 605k~751k ns로 나머지(158k~173k ns)보다 4배 이상 크고,
  같은 run들의 p999도 122k~123k ns로 다른 run(31k~67k)의 두 배 수준이다. sweep taker가 1,000개
  가격대를 매칭하는 동안 GC를 만나면 그 한 건이 통째로 부푸는 v0의 관찰과 동일한 패턴이다.

## 참고·한계

- v0-initial과 비교 시 측정 인스턴스·환경은 동일(루트 README 실행 환경)하지만 서로 다른 시점의
  단발 측정(n=10)이라 ±수% 차이는 노이즈와 실제 개선이 섞여 있을 수 있다. 워밍업의 실제 효과는
  `jit_compile_ms`처럼 워밍업이 직접 통제하는 신호(균일하게 낮아짐)에서 가장 분명하게 드러난다.
- per-run 값(run_0..run_9)은 원본 CSV에 있다. 위 이상치(run_6, run_3/7/8 등)는 이 CSV에서 직접
  확인한 값이다.
